\begin{frame}{Benchmark Framework Overview}\footnotesize
    The framework is organized as a reproducible pipeline: the same task set can be tested with different prompts, protocols and model backends.

    \vspace{0.35cm}
    \centering
    \texttt{tasks + prompts + protocols + model registries}

    \vspace{0.18cm}
    $\Downarrow$

    \vspace{0.18cm}
    \texttt{run\_benchmark.py}

    \vspace{0.18cm}
    $\Downarrow$

    \vspace{0.18cm}
    \texttt{runner/run\_suite.py} $\rightarrow$ \texttt{runner/run\_case.py}

    \vspace{0.18cm}
    $\Downarrow$

    \vspace{0.18cm}
    \texttt{model adapter} $\rightarrow$ \texttt{parser} $\rightarrow$ \texttt{VAL validator} $\rightarrow$ \texttt{metrics}

    \vspace{0.18cm}
    $\Downarrow$

    \vspace{0.18cm}
    \texttt{outputs/raw + outputs/parsed + outputs/scored}
\end{frame}

\begin{frame}{Why This Structure}\footnotesize
    \begin{itemize}
        \item \textbf{Comparable runs:} every model is evaluated on the same task, protocol and validation matrix.
        \item \textbf{Backend isolation:} model adapters keep HuggingFace, NVIDIA API, Ollama and llama.cpp behind the same generation interface.
        \item \textbf{External validation:} VAL checks the generated plans independently from the model text.
        \item \textbf{Traceable artifacts:} raw generations, parsed plans and scored validation results are stored separately.
        \item \textbf{Post-run analysis:} reports and plots are generated from saved JSON artifacts, not from manual inspection.
    \end{itemize}
\end{frame}

    \vspace{0.1cm} % Spazio elegante tra la parola e la linea di sottolineatura

    \textcolor{black}{\rule{10cm}{1pt}}
    
\end{frame}

\begin{frame}{Classification Purpose}\footnotesize
    \textcolor{blue}{\textbf{Purpose:}} select a small but meaningful set of PDDL tasks from the available domains.

    \vspace{0.35cm}
    The classification has two complementary goals:
    \begin{itemize}
        \item assign a numerical difficulty score to each domain and instance;
        \item label each domain by its dominant reasoning type.
    \end{itemize}

    \vspace{0.25cm}
    This makes the benchmark easier to interpret: model failures can be compared by both difficulty and planning style.
\end{frame}

\begin{frame}{Implementation: Raw and Numerical Score}\footnotesize
    Difficulty scores are computed with \texttt{Benchmark\_Framework/scripts/score\_domains\_complexity.py}.

    \vspace{0.25cm}
    The score combines features that increase planning complexity:
    \begin{multicols}{2}
    \begin{itemize}
        \item number of objects
        \item possible grounded actions
        \item number of goals
        \item numeric goal conditions
        \item numeric functions
        \item numeric preconditions
        \item numeric effects
        \item metric optimization
        \item topology structure
    \end{itemize}
    \end{multicols}

    Each instance receives a raw score, then scores are normalized within the domain and summarized at domain level.
\end{frame}


\begin{frame}{Implementation: Raw and Numerical Score II}\footnotesize
    The raw score is a weighted logarithmic combination, so no single count dominates the result:
    \resizebox{\textwidth}{!}{
        $RawScore = 0.45 \cdot \log(1 + Nactions) + 0.20 \cdot \log(1 + Ngoals) + 0.20 \cdot \log(1 + NumScore) + 0.15 \cdot Topology$
    }

    \vspace{0.2cm}
    \resizebox{\textwidth}{!}{
        $NumScore = 1.0 \cdot NumEffects + 2.0 \cdot NumPreconditions + 3.0 \cdot NumGoalConditions + 1.0 \cdot HasMetric$
    }

    \vspace{0.2cm}
    \begin{columns}
        \begin{column}{0.47\textwidth}
            The example shows the \textit{counters} domain summary: instance scores plus aggregate raw-score statistics.
        \end{column}
        
        \begin{column}{0.47\textwidth}
            \includegraphics[width=\textwidth]{domainsum.png}
        \end{column}
    \end{columns}
\end{frame}


\begin{frame}{Implementation: Reasoning Type}\footnotesize
    Difficulty alone is not enough: two domains can be equally hard while requiring different reasoning skills.

    \vspace{0.25cm}
    We therefore assign each domain a dominant reasoning category:
    \begin{multicols}{2}
    \begin{itemize}
        \item numerical optimization
        \item temporal planning
        \item logistics
        \item symbolic logic
        \item spatial planning
    \end{itemize}
    \end{multicols}

    Mixed domains are labeled by the reasoning type that is most central to solving the problem.
\end{frame}

\begin{frame}{Problems Chosen}\footnotesize
    We selected six domains from the three reasoning categories with enough coverage: numerical optimization, logistics and spatial planning.

    \vspace{0.25cm}
    \begin{columns}
        \begin{column}{0.47\textwidth}
            \centering \textbf{Hard Set}
            \begin{itemize}
                \item settlersnumeric (numerical optimization)
                \item rover (logistics)
                \item block-grouping (spatial)
            \end{itemize}
        \end{column}
        
        \begin{column}{0.47\textwidth}
            \centering \textbf{Easy Set}
            \begin{itemize}
                \item fo-counters (numerical optimization)
                \item expedition (logistics)
                \item fo-sailing (spatial)
            \end{itemize}
        \end{column}
    \end{columns}

    \vspace{0.25cm}
    Temporal planning and symbolic logic were excluded from the final set because only one domain was available for each category.
\end{frame}

\begin{frame}{Tools Used}\footnotesize
    The benchmark relies on a small set of tools, each with a specific role:
    \begin{itemize}
        \item \textbf{Leonardo HPC by CINECA:} main compute platform for local model execution.
        \item \textbf{Hugging Face:} source for local inference setup and download of open-source models.
        \item \textbf{whichLLM:} model-selection support based on available hardware on Leonardo.
        \item \textbf{VAL validator:} external PDDL plan validator.
        \item \textbf{NVIDIA API:} alternative backend to use open-source models without a local powerful machine.
    \end{itemize}

    \vspace{0.2cm}
    Both Leonardo and the NVIDIA API were used: Leonardo for controlled local execution with direct GPU access, and NVIDIA API as a hosted backend for additional model runs.
\end{frame}

\begin{frame}{Output Structure}\footnotesize
    Each benchmark case produces three aligned JSON artifacts:
    \begin{itemize}
        \item \textbf{Raw:} prompt messages, model output and provider-side reasoning text when available.
        \item \textbf{Parsed:} extracted PDDL actions, final plan text and formatting issues.
        \item \textbf{Scored:} VAL validation result, solved flag, iterations used and repair feedback.
    \end{itemize}

    \vspace{0.25cm}
    The suite also writes \texttt{suite\_result.json}, which summarizes the run matrix and is the entry point for post-run analysis.

    \vspace{0.25cm}
    This split keeps generation, extraction and evaluation independent, so parser errors and planning failures can be diagnosed separately.
\end{frame}

\begin{frame}{Evaluation Metrics Scope}\footnotesize
    The metrics are designed to answer three different questions:
    \begin{itemize}
        \item \textbf{Did the model solve the planning task?}
        \item \textbf{Was the produced action sequence grounded in the PDDL domain?}
        \item \textbf{Did retries improve the answer or only resample attempts?}
    \end{itemize}

    \vspace{0.25cm}
    For this reason, the evaluation does not rely only on final success. It combines success, executability, hallucination, precondition awareness and retry behavior.
\end{frame}

\begin{frame}{Core Metrics}\footnotesize
    \begin{itemize}
        \item \textbf{SR:} fraction of instances solved by the model.
        \item \textbf{FASR:} first-attempt success rate; measures zero-shot planning ability.
        \item \textbf{IWSR:} iteration-weighted success rate; rewards valid plans found with fewer retries.
        \item \textbf{Retry Gap:} $SR - FASR$; high values mean the model depends on repair attempts.
        \item \textbf{Executability Ratio:} fraction of the plan executable before the first precondition failure.
        \item \textbf{Hallucination Rate:} fraction of actions or objects not grounded in the PDDL files.
        \item \textbf{PAS:} precondition awareness score; separates sequencing errors from fabricated states.
        \item \textbf{CoTaL:} alignment between reasoning actions and the final extracted plan when reasoning is available.
    \end{itemize}
\end{frame}


\begin{frame}{Cross-Domain and Cross-Model Analysis}\footnotesize
    \begin{itemize}
        \item \textbf{Within-domain ranking:} compares models inside each domain using normalized metric values.
        \item \textbf{Cross-domain consistency:} checks whether a model is robust across different reasoning categories.
        \item \textbf{Failure mode taxonomy:} combines hallucination, executability and PAS to identify why plans fail.
        \item \textbf{Difficulty analysis:} compares easy, medium and hard instances to see where performance degrades.
    \end{itemize}

\end{frame}

\begin{frame}{Composite Planning Score}\footnotesize
    The Composite Planning Score summarizes complementary planning signals in one value:

    \vspace{0.2cm}
    \resizebox{\textwidth}{!}{
        $PS = 0.25 \cdot FASR + 0.20 \cdot IWSR + 0.20 \cdot ER + 0.20 \cdot (1 - Hallucination) + 0.15 \cdot PAS$
    }

    \vspace{0.35cm}
    \begin{itemize}
        \item \textbf{FASR} has the highest weight because it cannot be inflated by retries.
        \item \textbf{IWSR} and \textbf{ER} capture repair efficiency and structural plan quality.
        \item \textbf{Grounding} is represented by inverse hallucination rate.
        \item \textbf{PAS} adds information about local precondition reasoning.
    \end{itemize}
\end{frame}

\begin{frame}{Conclusions}\footnotesize
    \begin{itemize}
        \item The benchmark provides a reproducible way to compare LLMs on planning tasks, using the same domains, protocols and validator.
        \item VAL validation keeps final scoring independent from model wording and parser interpretation.
        \item The raw / parsed / scored split makes failures easier to inspect and compare.
        \item The metrics separate final success, grounding, executability, retry dependence and reasoning alignment.
        \item Future work can extend the task set, add more temporal and logic domains, and compare additional model backends.
    \end{itemize}
\end{frame}
